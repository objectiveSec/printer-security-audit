# printer-security-audit

Небольшой разбор инцидента с сетевым HP-принтером в доменной сети.

## Что произошло

В офисе принтер начал печатать много страниц с непонятным содержимым.  
В логах принтера повторялись задания примерно такого вида:

```text
user="Guest"
source_IP="0.0.0.0"
```

На первый взгляд это может выглядеть как неизвестный пользователь или кто-то посторонний в сети.  
Но по факту такой вывод делать нельзя. В подобных случаях `Guest` и `0.0.0.0` чаще означают, что принтер не смог нормально определить источник задания.

Самые вероятные причины:

- печать напрямую на IP принтера;
- RAW / JetDirect / Port 9100;
- зависшая очередь печати на одном из ПК;
- битый драйвер;
- некорректный PCL/PostScript-поток.

## Зачем этот репозиторий

Я сделал простой скрипт и чеклист, чтобы не разбирать такие случаи вручную каждый раз.

Скрипт не “ищет хакеров”.  
Он просто помогает быстро найти в логах признаки проблемной печати:

- задания от `Guest`;
- `source_IP="0.0.0.0"`;
- повторяющиеся задания;
- пустые или странные имена заданий;
- большое количество print events.

## Почему это относится к безопасности

Проблема не только в бумаге и тонере. Если принтер принимает задания напрямую от рабочих станций, то:

- сложно понять, кто реально отправил задание;
- печать может идти в обход print server;
- логи становятся бесполезными для расследования;
- любой хост в сети может слать данные на принтер, если доступ не ограничен;
- старые протоколы вроде RAW/9100 обычно не дают нормальной авторизации.

Это больше про hardening инфраструктуры, чем про “взлом”.

## Как запустить

```bash
python src/analyze_hp_logs.py examples/sample_hp_event_log.txt --out reports/sample_report.md
```

После запуска появится отчёт:

```text
reports/sample_report.md
```

## Что я бы проверял в реальной сети

1. Печатают ли пользователи напрямую на IP принтера.
2. Открыт ли Port 9100 / RAW / JetDirect.
3. Идут ли задания через print server.
4. Есть ли на ПК зависшие очереди печати.
5. Повторяются ли в логах `Guest` и `0.0.0.0`.
6. Ограничен ли доступ к принтеру по ACL/firewall.
7. Не включены ли лишние сетевые сервисы на принтере.

## Что делать при похожем случае

Короткий план:

1. Остановить текущую печать.
2. Сохранить Job Log / Event Log.
3. Проверить, есть ли `Guest` / `0.0.0.0`.
4. Проверить очереди печати на ПК и print server.
5. Запретить прямую печать на принтер, если она не нужна.
6. Оставить печать через управляемый print server.
7. Отключить RAW/9100 или ограничить его только для print server.

## Ограничения

Этот проект не доказывает наличие злоумышленника в домене.  
Он помогает разобрать один конкретный класс проблем: неконтролируемая или сломанная сетевая печать.

Нормальная формулировка результата:

> Найдены признаки неконтролируемой прямой печати и некорректных print jobs. Рекомендуется ограничить прямой доступ к принтеру и направить печать через print server.
